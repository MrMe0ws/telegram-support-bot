# Подсказки для агентов (Cursor / AI)

Краткий контекст репозитория, чтобы не ломать архитектуру и не раздувать диффы.

## Назначение

Telegram-бот поддержки: **ЛС пользователя ↔ топики форум-группы**, ответы саппорта в группе **копируются** клиенту в ЛС. БД — **SQLite**. Тексты интерфейса — **YAML** в `texts/<локаль>/`, доступ через `app.i18n.tr("файл", "ключ", ...)`.

## Стек

- Python 3.12, **aiogram 3**, SQLAlchemy 2, aiosqlite  
- Точка входа: `app/main.py` (polling, роутеры подключаются в явном порядке)

## Важные пути

| Путь | Содержимое |
|------|------------|
| `app/handlers/private_chat.py` | ЛС: старт, меню, пересылка в топик, `/close`, FSM «обращение» |
| `app/handlers/group_topics.py` | Группа: ответы саппорта → ЛС, `/close` и `/profile` в теме |
| `app/handlers/admin_panel.py` | Админка в ЛС (инлайн + FSM) |
| `app/handlers/help_callbacks.py` | Колбэки раздела «Помощь» |
| `app/handlers/message_reactions.py` | Зеркалирование реакций ЛС ↔ топик |
| `app/services/reaction_bridge.py` | Связки `dm_staff` и `group ↔ dm` для реакций |
| `app/services/tickets.py` | Тикеты; открытый тикет ищется по `user_id` **и** `source` (`MessageSource`) |
| `app/services/ticket_flow.py` | Закрытие тикета, уведомления |
| `app/db/models.py` | Модели; **`MessageSource`** (telegram/vk/max), **`Ticket.external_uid`**, **`ExternalSource`** — заготовка под другие каналы |
| `texts/ru/*.yml` | Все пользовательские строки; после правок нужен перезапуск бота |
| `texts/ru/help_links_seed.yml` | Посев пунктов «Помощь» (один раз при пустой таблице): поля `url` и/или `body_text` |

## Чего не трогать без нужды

- Каталог **`TelegramSupportBot-master/`** — сторонний референс (telebot, только Telegram, без форум-топиков), не используется рантаймом; для VK/MAX оттуда нечего переносить как готовый код.

## Внешние каналы (VK, MAX) — если появится задача

Сейчас в прод-потоке только **Telegram**. Для VK/MAX понадобятся отдельно:

1. **Учётные данные и конфиг** — токены/ключи сообщества VK, токен бота MAX, URL webhook или режим long polling (по документации API).
2. **Входящий адаптер** — HTTP endpoint (callback) и/или фоновый цикл, разбор событий, нормализация в «текст/медиа + id пользователя».
3. **Сопоставление с тикетом** — создать/найти `Ticket` с `source=vk|max`, заполнить `user_id` (или `external_uid`) стабильным id из API; создать топик в `SUPPORT_GROUP_ID` через тот же `Bot` Telegram (как в ЛС-флоу).
4. **Исходящий мост из топика** — в обработчике ответа саппорта учитывать `ticket.source`: если не `telegram`, вызывать VK/MAX API вместо `send_message` в Telegram ЛС.
5. **Ограничения UX** — админ-панель и «Помощь» завязаны на Telegram ЛС; для пользователей только VK/MAX понадобятся тексты/кнопки в их API или упрощённый сценарий без полного паритета.

Имеет смысл выделить общий слой «доставка клиенту» (интерфейс + реализации Telegram / VK / MAX), чтобы не дублировать логику топиков.

## Договорённости по коду

- Не менять тексты «для красоты» вне задачи; новые строки — в YAML и через `tr()`.  
- Не добавлять лишние зависимости без запроса.  
- Docker: образ `python:3.12-slim`; в `docker-compose.yml` заданы лимиты RAM (см. README).

## Ресурсы

При малом трафике процесс обычно укладывается в сотни мегабайт RAM; лимит **256 MiB** в compose — разумный запас для этого стека.
